### 1. Rate Limit 테스트 패턴 확립 (가장 중요!)

**문제 발생:**
- Feature 3.3.2, 3.3 Integration, Block 3 Module에서 반복적으로 rate limit 테스트 실패
- 테스트에서 `new Date()` 사용 → 현재 시간으로 record 생성 → 즉시 rate limit 걸림

**사용자 피드백:**
- Rate limit 로직 때문에 테스트가 현재 시간에 민감함

**핵심 원칙 (반드시 준수):**
```typescript
// ❌ 잘못된 방식:
const existingRecords = [
  { email: 'user@example.com', time: 100000, date: new Date() }
]

// ✅ 올바른 방식:
const twoMinutesAgo = new Date(Date.now() - 120000)
const existingRecords = [
  { email: 'user@example.com', time: 100000, date: twoMinutesAgo }
]
```

**표준 패턴:**
```typescript
// 모든 Integration/Module 테스트에서 사용
const twoMinutesAgo = new Date(Date.now() - 120000)

// 기존 record는 항상 과거 시간
storage.saveRecord({
  email: 'user@example.com',
  time: 100000,
  date: twoMinutesAgo // ✅ rate limit 회피
})

// 새로운 submission만 현재 시간 (validation 통과 보장)
const submission = recordSubmission.createSubmission(email, newTime)
```

**적용 범위:**
- Feature 3.3.2 (Record Validation) 테스트
- Feature 3.3 Integration Test
- Block 3 Module Test
- Product Integration Test

### 2. Multiple Elements 처리 전략

**문제 발생:**
- Feature 3.3 Integration: `Found multiple elements with the text: /1/`
- Block 3 Module: `Found multiple elements with the text: /rank/i`
- 원인: `/1/` 정규식이 "01:30.000"의 "1"과 rank "1" 모두 매칭

**해결 전략:**

**1. 숫자/짧은 텍스트 단독 검색 지양**
```typescript
// ❌ 잘못된 방식
expect(screen.getByText(/1/)).toBeInTheDocument()

// ✅ 올바른 방식
expect(screen.getByText(/rank/i)).toBeInTheDocument()
```

**2. 이미 검증된 값은 UI에서 재검증 불필요**
```typescript
// ✅ 로직 검증
expect(rank).toBe(1)
expect(screen.getByText(/rank/i)).toBeInTheDocument() // UI 존재만 확인

// ❌ 불필요한 중복 검증
// expect(screen.getByText(/1/)).toBeInTheDocument()
```

**3. 중복 예상되면 `getAllBy*` 사용**
```typescript
// Block 3 Module: RecordDisplay + LeaderboardTable
expect(screen.getAllByText(/rank/i)).toHaveLength(2) // h3 + th
expect(screen.getAllByText(/01:35.000/)).toHaveLength(2)
```

### 3. Validation 로직 단순화

**문제:**
- 초기 구현: Duplicate check와 improvement check 분리
- 테스트 실패: `expected 'Duplicate submission detected' to contain 'improvement'`

**해결:**
```typescript
// ❌ Before (복잡)
if (userRecord.time === time) {
  return { error: 'Duplicate submission detected' }
}
if (time >= userRecord.time) {
  return { error: 'No improvement over existing record' }
}

// ✅ After (단순)
// Combined: same time = no improvement
if (time >= userRecord.time) {
  return { error: 'No improvement over existing record' }
}
```

**장점:**
- 코드 줄 수 감소
- 에지 케이스 제거
- 비즈니스 로직 명확 ("개선만 허용")
